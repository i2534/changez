    不会冲突。这就是 git 的正常用法：


    本地 commits（没 push）          远程 commits
    ─────────────────              ─────────────────
    C3 ─ C4 ─ C5  (你的本地变更)     C1 ─ C2  (远程)
             ▲

    关键点：

    - 不 push = 不冲突 — 本地 commit 再多，只要不 push，远程毫不知情
    - pull 时才可能冲突 — 只有你 pull 远程新代码，且改了同一文件的同一行，git 才会报 conflict
    - 分支隔离 — 甚至可以在本地开独立分支，完全不影响主分支和远程

    典型做法：
    - 不想推的改动 commit 到本地，不 push 就行
    - 或者用 git stash 临时搁置
    - 或者开本地分支 git checkout -b local-work

    所以不用担心，你完全可以放心用本地 git 管理那些不想上远程的变更。changez 在这之上再加一层记录，三者互不干扰。

+构建一个轻量级文件版本追踪服务，为不在 git 管理中或不方便 push 的本地变更提供独立备份和恢复能力。
+
 - **记录**文件的创建、修改、删除操作
-- **区分**变更来源：AI 工具（Cursor / OpenCode / Claude Code）vs 人工编辑
-- **空间优化**的版本存储（delta 压缩 + 内容寻址去重）
-- 提供查询/审计界面
+- **标记**变更来源：AI 工具（Cursor / Claude Code / OpenCode）或人工

 ### 1.2 设计原则

 | 原则 | 说明 |
 |------|------|
-| 非侵入式 | 不修改 AI 工具源码，通过 hook / FS watcher / 进程探测被动采集 |

-
-#### Aider (⭐ 26k+)
-- **原理**: 在 git 仓库中工作，AI 每次编辑后自动 commit，commit message 带 `aider:` 前缀
-- **AI 归因**: 通过 commit message 前缀区分
-- **版本存储**: 完全依赖 git 的 delta 压缩
-- **优点**: 成熟稳定，与 git 无缝集成
-- **缺点**:
-  - 只能追踪 Aider 自己的变更，无法跨工具
-  - commit 粒度由 Aider 控制，无法统一视角
-  - 人工编辑和 AI 编辑混在同一 git history 中，需正则匹配区分
-
-#### Claude Code
-- **原理**: Agent 模式运行，每次 session 结束后可选择将变更 commit
-- **AI 归因**: session 日志 + commit，但 commit 不强制标记来源
-- **版本存储**: 依赖 git
-- **优点**: 有完整的 session 日志可追溯
-- **缺点**:
-  - 需要手动开启 `--output-session`
-  - 无法区分同一 session 内哪些文件是 AI 改的
-  - 不支持跨工具统一追踪
-
-#### Cursor
-- **原理**: IDE 内置 AI，直接修改文件
-- **AI 归因**:
-  - Cursor 有内置的 "Changes" 面板显示 AI 编辑历史
-  - 但这是 IDE 内部的，无法导出为结构化数据
-  - 没有公开的 API 或 hook 机制
-- **版本存储**: IDE 内部 + git
-- **缺点**:
-  - 黑盒，无法外部接入
-  - 只能通过 FS watcher 检测文件变更，再结合进程上下文推断来源
-  - 无法 100% 确定变更来源
-
-#### OpenCode
-- **原理**: 终端 AI 编码 agent，类似 Claude Code
-- **AI 归因**: 通过进程监听 + session 日志
-- **版本存储**: 依赖 git
-- **优点**: 开源，可接入
-- **缺点**: 同样需要外部 hook 机制
-
-#### Cline (VS Code Extension)
-- **原理**: VS Code 扩展，AI 编码 agent
-- **AI 归因**: 有内置的 changelog 功能，记录每个 AI 操作
-- **版本存储**: 依赖 git
-- **优点**: 有结构化的操作日志

● sqlite存在的作用是什么?
────────────────────────────────────────


╭─ ⚕ Hermes ───────────────────────────────────────────╮
    好问题。在这个场景下，SQLite 管的是元数据和索引，blob 管的是文件内容。

    具体来说 SQLite 回答这些问题：

    - "文件 X 有多少个版本？分别是什么时候，谁改的？" → 查 versions 表
    - "昨天 opencode 改了哪些文件？" → SELECT ... WHERE source='opencode' AND changed_at > ...
    - "文件 X 第 42 行最后是谁改的？" → 按版本倒序查 change_summaries
    - "版本 40 的内容存在哪个 blob？" → versions.blob_hash → blobs/40/ef1234...

    如果没有 SQLite，纯靠文件系统：
    - 每个文件一堆快照文件，得遍历目录 + 解析文件名来排序/过滤
    - 按时间、来源、文件类型查历史 → 全量扫描，慢
    - 多版本并发写 → 没有事务，容易丢数据

    但确实可以简化。 如果你觉得 SQLite 太重，替代方案：

    | 方案          | 优点                                       | 缺点                     |
    |---------------|--------------------------------------------|--------------------------|
    | SQLite (当前) | 查询快，有事务，纯 Go (modernc.org/sqlite) | 一个依赖                 |
    | BadgerDB (KV) | 你 notice 项目已经在用，熟悉               | 复杂查询不如 SQL         |
    | 纯 JSON 索引  | 零依赖                                     | 并发写有问题，大数据量慢 |
    | BoltDB (KV)   | 纯 Go，轻量                                | 同样复杂查询弱           |

    实话讲，对于这个用途 SQLite 是最省心的——一个嵌入式文件，不用起额外服务，modernc.org/sqlite 纯 Go 无 CGO。但你有偏好的话我可以换。

    你觉得 SQLite 可以接受，还是想换个方案？

╭─ ⚕ Hermes ───────────────────────────────────────────╮
    具体方案：

    1. 文件命名

    SHA-256 hash，2 字符前缀分目录（防单目录文件过多）：


    blobs/
      ab/
        cdef1234567890abcd...    # 完整 64 字符 hash 作为文件名
      ff/
        1234567890abcdef...


    每个 blob 文件 = zstd.Compress(originalContent)，解压即得原始内容。

    2. 版本和 blob 的关系


    versions 表:
      id=40, file_id=1, blob_hash="abcdef...", delta_from_id=NULL   → 完整快照
      id=41, file_id=1, blob_hash="123456...", delta_from_id=NULL   → 完整快照
      id=42, file_id=1, blob_hash=NULL,     delta_from_id=40, delta_blob_hash="789abc..."  → GC 后转 delta


    - 正常写入：每个版本存完整内容到 blobs，delta_from_id=NULL
    - GC 后：老版本的完整 blob 被替换成 delta 指针，delta_blob_hash 指向 deltas/ 下的 delta 文件

    3. Delta 存哪

    这里有个选择：

    | 方案 | 结构                           | 说明                                            |
    |------|--------------------------------|-------------------------------------------------|
    | A    | deltas 放 blobs 里（同 store） | blobs/delta-789abc...，一套 CAS 管到底          |
    | B    | 独立 deltas/ 目录              | deltas/78/9abc...，快照和 delta 物理分离        |
    | C    | delta 内联存 SQLite            | BLOB 列直接存 delta 数据，小 delta 不单独建文件 |

    我倾向 A（统一 CAS）或 C（小 delta 放数据库，减少文件碎片）。你倾向哪种？

    另外 delta 格式也用 xdelta（字节级）还是行级 diff（类似 diff -u patch）？文本文件的话行级 diff 更省空间也更易读。

    行级 diff 方向没问题。几个细节可以敲定：

    Delta 存储位置（三选一）

    | 方案                     | 优点                                  | 缺点                                |
    |--------------------------|---------------------------------------|-------------------------------------|
    | A. deltas/ 独立目录      | 物理分离，清晰                        | 多一套目录管理                      |

    行级 diff 通常很小（几行到几十行），我倾向 C（内联存 SQLite），理由：
    - 减少文件系统碎片（一个文件的几百个版本不产生几百个小文件）
    - 读的时候一次 SQL 拿到元数据 + delta，不需要再去读文件
    - SQLite 本身有 WAL 模式，写性能不是问题

    Delta 格式（行级）


    // 方案 1: 标准 unified diff (patch)
    --- a/file.go
    +++ b/file.go
    @@ -10,3 +10,5 @@
     line10
    -line11
    +line11_modified
    +line11_new

    // 方案 2: 自定义紧凑格式
    [base_version_id]
    [op:keep|ins|del] [line_num] [content]
    keep 10 "line10"
    del  11 "line11"
    ins  11 "line11_modified"
    ins  12 "line11_new"


    方案 1 优势：标准格式，patch 命令直接能用，调试方便
    方案 2 优势：解析更快，无冗余文本

    好问题，一一回答：

    1. Delta 存 SQLite 的膨胀问题

    会。SQLite 的 BLOB 一旦写入，即使删除也不会自动收缩文件（只标记空闲页）。假设一个文件 100 个版本，每个 delta 平均 2KB，累积就是 200KB 进了数据库。文件多了数据库轻松上几十 MB。

    而且 SQLite 不适合存大量中大 BLOB——它不是对象存储，频繁写 BLOB 会导致数据库碎片化，需要定期 VACUUM。

    结论：delta 放独立 deltas/ 目录，SQLite 只存元数据和 hash 指针。


    SQLite: 轻量索引（几个 MB）
    blobs/:  完整快照（zstd 压缩）
    deltas/: delta patch（纯文本 unified diff，可读可调试）


    2. 连续相同内容的快照

    选 A) 跳过。逻辑：


    收到 snapshot → SHA-256(content) → 对比该文件最新版本
      相同 → 不写新版本记录，返回 "unchanged"
      不同 → 正常写入


    这样即使 AI 工具习惯性全量上报，也不会产生垃圾记录。只有真正变了才记。

    3. restore 先放一放

    好，先聚焦服务端核心：snapshot 上报 + 存储 + 查询。

● 对于1, 使用 blob和delta文件, 如果很多一两行的修改, 就会产生很多文件碎片吧
────────────────────────────────────────


╭─ ⚕ Hermes ───────────────────────────────────────────╮
    对，这是真问题。一个文件 100 次修改 = 100 个 blob 小文件 + 可能 80+ 个 delta 小文件。几千上万个 1KB 的文件对文件系统不友好。

    换个思路：文本文件不大，干脆直接存 SQLite BLOB。

    算笔账：
    - 一个 Go 源文件平均 5KB
    - 100 次修改 = 500KB
    - 1000 个文件 x 100 版本 = 500MB（SQLite，未压缩）
    - SQLite 支持 zlib 压缩或者我们入库前 zstd 压缩 = 可能只要 100MB

    混合方案：


    versions 表加一列:

      storage_mode TEXT  -- 'inline' | 'blob' | 'delta'
      inline_content BLOB  -- storage_mode='inline' 时直接用

    判断逻辑:
      - 内容 < 64KB → inline，直接存 SQLite BLOB（入库前 zstd 压缩）
      - 内容 >= 64KB → 存 blobs/ 文件
      - GC 时老版本转 delta → 存 deltas/ 文件


    这样：
    - 90%+ 的代码文件 < 64KB，走 inline，零文件碎片
    - 大文件走 blobs/，不影响
    - delta 只在版本很多时才有，数量可控
    - SQLite 体积可控（小文件压缩后很小）

    你觉得这个混合方案可行吗？阈值 64KB 合适还是调一下？

● 为什么我们不把delta变更写入到一个文件内?
────────────────────────────────────────


╭─ ⚕ Hermes ───────────────────────────────────────────╮
    好主意。一个追加写入的 delta 日志文件，类似 git packfile 的思路。


    deltas/
      delta.log          # 所有 delta 追加到一个文件


    Entry 格式：


    +--------+--------+------------------+
    | length (4B) | versionId (4B) | delta content (N bytes) |
    +--------+--------+------------------+


    SQLite 里记 offset + length 代替文件名：

    sql
    versions:
      storage_mode = 'delta'
      delta_offset = 18432    -- delta.log 中的偏移量
      delta_length = 256      -- 内容长度
      delta_base_id = 38      -- 基于哪个版本还原


    优点：
    - 零碎片，就一个文件
    - 纯追加写入，不需要 lock（读和追加不冲突）
    - 通过 offset 随机读取，O(1) 定位
    - 文件数量少，inode 不浪费

    注意：
    - 只追加不修改，所以不会缩小（但 delta 本来就小）
    - 读的时候 seek(offset) → 读 length 字节 → 解码

    这样三个存储模式就齐了：

    | mode   | 适用           | 存储位置                         |
    |--------|----------------|----------------------------------|
    | inline | 新快照，< 64KB | SQLite BLOB                      |
    | blob   | 新快照，≥ 64KB | blobs/ 独立文件                  |
    | delta  | GC 后的老版本  | deltas/delta.log (offset+length) |

    这个方案可以吗？还有别的细节要过的吗？

    少一张表，ON DELETE CASCADE 删文件时级联清版本。

    2.2 delta 的 base_id 指向谁？

    两种模式：


    模式 A: 链式（每个 delta 指向前一个版本）
    v1(blob) ← v2(delta, base=v1) ← v3(delta, base=v2) ← v4(delta, base=v3)
    还原 v4: read(v1) → apply(v2) → apply(v3) → apply(v4)  链式 3 步

    模式 B: 星型（每个 delta 指向最近的 blob checkpoint）
    v1(blob) ← v2(delta, base=v1) ← v3(delta, base=v1) ← v4(delta, base=v1)
    还原 v4: read(v1) → apply(v4)  只需 1 步


    |        | 空间                                       | 读取速度         |
    |--------|--------------------------------------------|------------------|
    | A 链式 | 省（每次只 diff 相邻版本，变更小）         | 慢（链越长越慢） |
    | B 星型 | 浪费（每次 diff 都是对比 base，diff 量大） | 快（一步到位）   |

    我倾向 A 链式（省空间），compact 截断链控读性能。

    1. 字节序 + magic number

    源代码/文本协议普遍用 大端（network byte order），因为 hex dump 时肉眼可读，跨平台一致。


    +--------+--------+--------+--------+----------+
    | Magic  | Version| length | versionId | isCompressed | delta content |
    | 4B     | 4B     | 4B     | 4B        | 1B           | N bytes      |


    - Magic: 固定 0x4348444C（"CHDL" = Changez Delta Log）
    - Version: 当前 0x00000001，以后格式升级加版本判断
    - length: delta content 的长度（不含 header，17B）
    - versionId: 对应的 versions.id
    - isCompressed: 0=明文, 1=zstd

    header 总共 17 字节。读的时候先读 17B header，校验 magic+version，再按 length 读 content。

    5. compact 双触发 ✅


    写入时: 检查该文件的 delta 链长，超过阈值 → 立即 compact 当前版本
    定时器: 定期检查所有文件，清理孤立 blob，处理写入时没赶上的




    2. 并发写入，展开讨论

    场景：OpenCode 和 Claude Code 同时在同一个项目里改文件，各自调 snapshot。

    | 情况           | 例子                         | 难度 |
    |----------------|------------------------------|------|
    | 同文件同时写   | 两个 IDE 同时改了 main.go    | 高   |

    1. Magic + version 合并 4B

    | Magic  | length | versionId | isCompressed | delta content |
    | 4B     | 4B     | 4B       | 1B           | N bytes      |

    2. zstd 的好处

    | 特性       | gzip               | bz2   | zstd            |
    |------------|--------------------|-------|-----------------|
    | 压缩率     | 中                 | 高    | 高（≈bz2）      |
    | 解压速度   | 快                 | 慢    | 最快            |
    | 压缩速度   | 中                 | 慢    | 快              |
    | 可调级别   | 1-9                | -     | 1-22            |
    | 小文件开销 | ~20B header+footer | ~16B  | ~19B            |
    | 内置校验   | CRC32              | CRC32 | 64-bit checksum |

    核心优势：解压速度比 gzip 快 2-3 倍，压缩率还更高。

    对这个项目特别重要：
    - 读版本时需要解压 blob 和 delta，解压快 = 查询响应快
    - 代码文件压缩后通常很小（5KB → 1-2KB），zstd 在小文件上表现也不错
    - Go 有成熟纯 Go 实现 github.com/klauspost/compress/zstd，无 CGO

    delta 内容通常很小（几行 diff 才几百字节），不压缩也才省 13B header。但统一用 zstd 逻辑简单——一个代码路径处理所有压缩，不用根据大小分叉。

● 对于方案A, 我还有疑问, 如果我某个IDE只想恢复它自己的修改, 可以跳过别人的delta恢复吗?
────────────────────────────────────────


╭─ ⚕ Hermes ───────────────────────────────────────────╮
    不能直接跳过。 Delta 链是累积的，像叠罗汉：


    v1(content) → v2(delta_A) → v3(delta_B) → v4(delta_C)


    v4 的内容 = v1 + delta_A + delta_B + delta_C。你不能只拿 v1 + delta_A + delta_C，因为 delta_C 是基于 (v1+A+B) 算出来的 diff，跳到上面格式对不上，patch 会失败。

    能做到的是：恢复到指定版本。


    "我要回退到 v2" → 还原 v1 + delta_A → 写回文件
    "我要回退到 v3" → 还原 v1 + delta_A + delta_B → 写回文件


    结合 source 信息，用户可以：


    changez log main.go
      v4  cursor     +8行  09:30
      v3  opencode   +15行 09:25
      v2  claude-code -3行  09:20
      v1  human      (base)

    "我只想回退 cursor 的改动" → restore to v3
    changez restore main.go --version=3


    做不到的是： cherry-pick（只保留某个 IDE 的改动，跳过中间的）。 那需要每个 delta 可逆 + 重新计算，复杂度上升一个量级。

    对于"文件版本追踪/兜底备份"这个定位，恢复到指定版本已经够用了。cherry-pick 是 git 级别的功能，不在当前范围内。

● cherry pick 是什么, 我们这里实现需要做哪些工作?
────────────────────────────────────────


╭─ ⚕ Hermes ───────────────────────────────────────────╮
    Cherry-pick = 挑选性恢复。 从历史中挑几个版本的改动应用，跳过其他的。


    历史: v1(base) → v2(opencode) → v3(claude-code) → v4(cursor)

    普通 restore: 回到 v3 = 应用 v2+v3 的改动
    Cherry-pick:  只要 opencode 和 cursor = 应用 v2+v4，跳过 v3


    实现需要的额外工作：

    第 1 步已经做了：任意版本能还原完整内容（blob + 链式 apply delta）。

    缺的是：


    额外工作 1: 可逆 delta
      当前: delta 是 forward patch (v2→v3)
      需要: 能反向 apply (v3→v2)，或者重新生成 reverse diff
      工作量: unified diff 天然可逆（交换 ---/+++ 方向），解析现有 diff 反转即可

    额外工作 2: 中间重新 diff
      场景: 要 cherry-pick v2+v4，跳过 v3
      问题: v4 的 delta 是基于 v3 算的，不能直接 applied 到 v2 的结果上
      需要: 还原 v2 完整内容 + 还原 v4 完整内容 → 重新算 diff → 应用
      工作量: 已经有 diff 库，多一次 encode 操作

    额外工作 3: API 设计
      changez_restore --version=3              ← 已有
      changez_restore --pick=2,4 --skip=3      ← 新增
      changez_restore --source=opencode        ← 按来源筛选


    总结：

    |                      | 工作量                         | 复杂度 |
    |----------------------|--------------------------------|--------|
    | 普通 restore（已有） | 链式 apply delta               | 低     |
    | cherry-pick          | + 全量还原参与版本 + 重新 diff | 中     |
    | 新增 API 参数        | --pick, --skip, --source       | 低     |

    核心成本在工作 2：cherry-pick 必须先还原涉及的所有版本的完整内容，再重新算 diff。对于大文件或长链，内存和 CPU 开销明显。

    我的建议：Phase 1 不做。 理由：
    - 当前定位是备份兜底，不是 git 替代品
    - 99% 的场景"恢复到某个版本"就够用了
    - 功能可以后续加，数据结构不需要改（已经支持任意版本还原）

    两个问题拆开看：

    1. 无 metadata 时省空间

    核心思路：不加固定的 metadata_length 字段，用 flag 指示有没有。


    方案 A: 借用 compressMethod 高位做 flag
      compressMethod 2B: [1B flags][1B method]
      flags bit0: hasMetadata

      无 metadata: header 14B（Magic+length+versionId+compressMethod）
      有 metadata: diff 后面直接跟 JSON，读到 EOF 或下一条 entry 前都是 metadata

    方案 B: 独立 flags 字节
      header 加 1B flags
      无 metadata: header 15B
      有 metadata: diff 后面跟 [4B meta_length][N bytes JSON]


    |     | 无 metadata header | 有 metadata header              | 扩展性                        |
    |-----|--------------------|---------------------------------|-------------------------------|
    | A   | 14B（不变）        | 14B + diff + JSON（无长度标记） | 弱（flags 位用完要改 header） |
    | B   | 15B（+1B）         | 15B + diff + 4B + JSON          | 强（7 个 flag bit 备用）      |

    方案 A 省 1B 但 metadata 没长度标记，解析时要靠下一条 entry 的 magic 来断界，复杂度高。

    2. metadata 要不要压缩

    metadata 当前就一条 session JSON，大概 50-100 字节：

    json

    zstd 压缩 100B 数据，header 开销 19B，压缩后可能反而更大。不压缩。

    如果未来 metadata 变大（比如加了 prompt 摘要），再考虑。flags 留一个 bit 做 compressedMetadata 就行。

+# Changez Design — Hermes 文件系统版本管理插件
+
+## 概述
+
+为 Hermes Agent 提供文件系统的版本管理能力：自动快照、历史回溯、版本恢复。
+核心思路：基于 git diff 的增量存储，不依赖 git 仓库。
+
+## 数据目录
+
+```
+~/.hermes/changez/
+  data/           # deltas + metadata（snake_case 命名）
+  wal/            # 预写日志
+  changez.db      # SQLite 数据库
+```
+
+## 数据库表
+
+### deltas
+
+存储文件增量的二进制内容，按文件路径分组。
+
+```sql
+CREATE TABLE deltas (
+  id           INTEGER PRIMARY KEY AUTOINCREMENT,
+  file_path    TEXT NOT NULL,
+  delta_type   TEXT NOT NULL DEFAULT 'full',   -- full | patch
+  previous_id  INTEGER,                        -- 前一个 delta 的 id（patch 模式）
+  content      BLOB NOT NULL,                  -- 压缩后的 diff 内容
+  hash         TEXT,                           -- SHA256 of uncompressed content
+  compressed   INTEGER DEFAULT 0,              -- 0=raw, 1=zstd
+  metadata     TEXT,                           -- JSON: source, sessionId, message 等
+  created_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
+);
+
+CREATE INDEX idx_deltas_file_path ON deltas(file_path);
+```
+
+- `delta_type`: `full`（首次快照存完整文件），后续用 `patch`
+- `previous_id`: patch 模式指向上一个 delta 的 id，用于重建
+- `content`: 14B header + 压缩后的 diff 内容
+- `metadata`: JSON 字符串，可选字段
+
+### versions
+
+索引版本，关联 delta 和元数据。
+
+```sql
+CREATE TABLE versions (
+  id           INTEGER PRIMARY KEY AUTOINCREMENT,
+  file_path    TEXT NOT NULL,
+  delta_id     INTEGER NOT NULL REFERENCES deltas(id),
+  hash         TEXT,                           -- SHA256 of reconstructed full content
+  source       TEXT,                           -- opencode | claude-code | cursor | human
+  session_id   TEXT,
+  message      TEXT,                           -- commit message（snapshot 时传入）
+  created_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
+);
+
+CREATE INDEX idx_versions_file_path ON versions(file_path);
+CREATE INDEX idx_versions_created_at ON versions(created_at);
+```
+
+## Delta 文件格式
+
+每个 delta 的 content 字段采用固定 header + 压缩内容的格式：
+
+```
++--------+------------+---------------+--------------+-------------+
+| magic  | versionId  | compressMethod| deltaLength  | metaLength  |
+| 4B     | 4B         | 2B            | 4B           | 4B          |
+```
+
+Header 固定 **14 字节**，所有字段 big-endian。
+
+- `magic`: `0x43440001` ("CD" + v1)
+- `versionId`: versions.id，用于校验
+- `compressMethod`: `0x0000`=无压缩, `0x0001`=zstd
